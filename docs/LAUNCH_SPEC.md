# Swipd — Spec-план запуска в продакшн

> Статус: черновик v1 · Контекст: сейчас это frontend-only MVP (Vite + React + Zustand),
> данные в `localStorage` и статических `.ts`, медиа в `public/`, хостинг — временный туннель.
> Цель документа: довести до боевого запуска (регистрация, загрузка контента, быстрый
> сервер, управляемость).

---

## 0. Constitution (принципы)

1. **Serverless-first.** Не держим и не масштабируем свой сервер вручную — используем
   managed-сервисы (БД, авторизация, хранилище, видео). «Быстрый сервер» = его отсутствие
   + CDN на краю сети.
2. **Медиа — отдельно от приложения.** Фото/видео никогда не лежат в бандле/репозитории —
   только в объектном хранилище + CDN, видео — через стриминговый сервис (HLS).
3. **Один источник правды — БД.** Каталог, заказы, пользователи переезжают из статики в
   Postgres. Фронт читает через API/SDK.
4. **Безопасность по умолчанию.** Доступ через Row-Level Security: продавец видит/меняет
   только своё, покупатель — только своё.
5. **Измеримость.** Каждый ключевой шаг воронки логируется (продуктовая аналитика), ошибки
   ловятся (мониторинг).

## 1. Что нужно к запуску (scope)

### 1.1 Регистрация и роли (БД)
- Регистрация по **email + OTP** и/или **телефон (OTP)**; вход через Google.
- Роли: **buyer** / **seller**. Продавец при регистрации создаёт магазин.
- Профиль, адреса, избранное, подписки, корзина — в БД, привязаны к пользователю.

### 1.2 Загрузка контента продавцом
- **Товарные карточки**: создать/редактировать/скрыть товар (название, цена, категория,
  цвета, размеры, описание, детали, стиль-теги).
- **Загрузка фото** (несколько на товар) → хранилище → авто-оптимизация (WebP, ресайз).
- **Загрузка видео** → стриминговый сервис → авто-транскод в HLS + постер-кадр → быстрый
  показ в ленте.
- Превью карточки в реальном времени (как будет выглядеть в Discovery / магазине).

### 1.3 Быстрый сервер (производительность)
- SPA на глобальном CDN (edge), а не на одном сервере.
- Видео — адаптивный битрейт (HLS), автопостеры, edge-доставка.
- Картинки — on-the-fly ресайз + WebP/AVIF через image-CDN.
- Только активное видео в ленте грузится (уже сделано на фронте — TikTok-режим).

### 1.4 Управление и контроль (ops)
- **Кабинет продавца**: товары, медиа, заказы со статусами, чат, аналитика магазина.
- **Админка платформы**: модерация магазинов/контента, верификация продавцов, выплаты,
  споры, глобальная аналитика.
- **Покупатель**: профиль, заказы, подписки, поддержка.

### Non-goals (не делаем на запуске)
- Собственный CDN/транскодер с нуля. Сложные ML-рекомендации. Мультиязычность.
  Маркетплейс-логистика (берём курьерку как есть).

## 2. Рекомендуемый стек (самый эффективный для скорости и сроков)

| Слой | Инструмент | Почему |
|---|---|---|
| **Хостинг фронта** | **Cloudflare Pages** или Vercel | Статика на глобальном CDN, мгновенно, авто-деплой из Git, бесплатно на старте |
| **БД + Auth + Storage + API** | **Supabase** (Postgres) | Один сервис закрывает регистрацию, товары, заказы, загрузку фото, RLS-безопасность, realtime. Минимум кода бэкенда |
| **Видео** | **Cloudflare Stream** (или bunny.net Stream) | Авто-транскод в HLS, постеры, адаптивный битрейт, edge-доставка → видео стартует быстро. Решает «долго грузится» |
| **Картинки** | **Cloudflare Images** / Supabase image transforms | WebP/AVIF, ресайз на лету, кэш на краю |
| **Платежи (KZ)** | **Kaspi Pay / CloudPayments / Robokassa** | Местные карты, Kaspi QR. Stripe — если будет международка |
| **Уведомления** | Telegram-бот (уже есть) + **Resend** (email) | Заказы продавцу в Telegram, письма покупателю |
| **Продуктовая аналитика** | **PostHog** | Воронки, события, session replay. Заменяет наш localStorage-логгер |
| **Мониторинг ошибок** | **Sentry** | Видеть фризы/краши в проде с трейсом |
| **Поиск (позже)** | Postgres FTS → **Meilisearch/Algolia** | Когда товаров станет много |

> Стоимость на старте: всё перечисленное имеет бесплатные/дешёвые тарифы. Реальные деньги
> появятся на видео-трафике (Stream считает по минутам просмотра) и платёжных комиссиях.

**Почему так «держит сервер быстрым»:** нет своего сервера, который надо тюнить. Фронт —
статика на CDN; данные — managed Postgres; медиа — стриминг/картиночный CDN. Всё
масштабируется автоматически и отдаётся с ближайшей точки к пользователю.

## 3. Модель данных (Supabase / Postgres)

```
profiles(id=auth.uid, name, username, avatar_url, role[buyer|seller], created_at)
stores(id, owner_id→profiles, name, slug, tagline, cover_url, verified, status)
products(id, store_id→stores, title, price, old_price, category, description,
         details jsonb, sizes text[], colors jsonb, style_tags text[],
         in_stock bool, rating numeric, created_at)
product_media(id, product_id→products, kind[image|video], url, poster_url,
              stream_uid, sort)
orders(id, buyer_id→profiles, store_id→stores, status, total,
       customer jsonb, delivery, payment, created_at)
order_items(id, order_id→orders, product_id, size, color, qty, price)
addresses(id, user_id→profiles, label, city, street, recipient, phone)
reviews(id, product_id, author_id, rating, text, photo_url, created_at)
follows(user_id→profiles, store_id→stores)         -- подписки
analytics_events(id, user_id, event, props jsonb, ts) -- или сразу PostHog
```

**RLS (примеры):** продавец делает CRUD только там, где `stores.owner_id = auth.uid()`;
заказы видны покупателю (`buyer_id`) и владельцу магазина; каталог читаем всем.

## 4. Поток загрузки медиа (без своего сервера)

1. Продавец в кабинете жмёт «Добавить видео».
2. Фронт просит у Supabase Edge Function **direct-upload URL** от Cloudflare Stream.
3. Браузер заливает файл **напрямую** в Stream (минуя наш бэкенд).
4. Stream транскодит → HLS + постер; webhook кладёт `stream_uid` и `poster_url` в `product_media`.
5. Фронт в ленте играет HLS активного видео (уже есть TikTok-режим воспроизведения).

Фото — аналогично: signed upload в Supabase Storage → отдаём через image-CDN с ресайзом.

## 5. Анализ пути клиента + модель управления

### Воронка (и где контроль/инструмент)
| Этап | Что происходит | Контроль / инструмент |
|---|---|---|
| **Привлечение** | Ссылка/Instagram/реклама | UTM-метки, PostHog acquisition |
| **Первый вход** | Онбординг + регистрация | Supabase Auth, % завершения онбординга |
| **Discovery** | Лента, свайпы, видео | feed_view/swipe события, скорость видео |
| **Вовлечение** | Лайк, сохранить, подписка | follows, style-preferences |
| **Товар** | Карточка, размер, цвет | product_opened, добавления |
| **Корзина → Checkout** | Оформление, оплата | checkout_started→order_completed, платёжка |
| **Фулфилмент** | Продавец собирает заказ | Кабинет продавца, Telegram-уведомление |
| **Доставка** | Курьер/ПВЗ | Статусы заказа |
| **После покупки** | Отзыв, повтор | reviews, push/email |
| **Удержание** | Подписки, новинки | Лента «Подписки», рассылки |

### Кто чем управляет (3 роли)
- **Покупатель** — профиль, заказы, подписки, поддержка.
- **Продавец (кабинет)** — товары, загрузка медиа, заказы + статусы, чат с ИИ/клиентом,
  аналитика магазина, выплаты.
- **Админ платформы** — модерация магазинов и контента, верификация продавцов («официальный
  магазин» галочка), управление выплатами и спорами, глобальные метрики, фиче-флаги.

**Самый эффективный вариант ведения:** один **Seller Dashboard** (расширяем текущий
«Кабинет продавца») + отдельная **Admin-консоль** поверх той же БД. Не плодить отдельные
системы — всё на Supabase, роли разводятся через RLS. Операционка (заказы, споры,
модерация) — там же, аналитика — в PostHog, инциденты — в Sentry.

## 6. Дорожная карта (фазы)

- **Фаза 0 — Хардеринг (1–2 дня).** Деплой SPA на Cloudflare Pages, сжать текущие видео,
  вынести конфиг в env, подключить Sentry. → стабильная постоянная ссылка.
- **Фаза 1 — Бэкенд-фундамент.** Проект Supabase, схема БД, Auth (buyer/seller), миграция
  статического каталога в БД, RLS. → регистрация и данные «живые».
- **Фаза 2 — Загрузка контента.** Supabase Storage + Cloudflare Stream, CRUD товаров и
  медиа в кабинете продавца, превью карточки. → продавцы заливают свой контент.
- **Фаза 3 — Коммерция.** Реальные заказы в БД, оплата (Kaspi/CloudPayments), статусы,
  уведомления (Telegram + email). → можно принимать деньги.
- **Фаза 4 — Управление.** Admin-консоль (модерация, верификация, выплаты), PostHog-воронки.
  → платформой можно управлять.
- **Фаза 5 — Масштаб/полировка.** Поиск, серверные рекомендации, тюнинг image/video CDN,
  SEO/PWA, нагрузочные тесты.

## 7. Критерии готовности к запуску (Definition of Done)

- [ ] Регистрация работает: покупатель и продавец, роли разведены.
- [ ] Продавец загружает товар + видео → появляется в ленте и в своём магазине.
- [ ] Покупатель оформляет заказ и **оплачивает**; продавец видит заказ + Telegram-уведомление.
- [ ] Сайт грузится < 2 c на 4G; видео стартует < 1 c (HLS).
- [ ] Постоянный домен на CDN, без туннеля.
- [ ] Sentry ловит ошибки, PostHog пишет воронку.
- [ ] Базовая модерация: админ может скрыть товар/магазин.
- [ ] Бэкапы БД включены, секреты в env, RLS закрывает чужие данные.

---

## 8. Риски

- **Видео-трафик дорогой** при росте просмотров (Stream по минутам) — мониторить, сжимать,
  ограничивать автоплей.
- **Платежи в KZ** требуют договора с провайдером (Kaspi/CloudPayments) — заложить время.
- **Модерация UGC** (загрузка продавцами) — нужен хотя бы ручной разбор жалоб с первого дня.
- **Миграция со static → DB** — держать фронт обратно совместимым (адаптер данных).
